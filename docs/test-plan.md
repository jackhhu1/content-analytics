## Unit Tests (Vitest)

### lib/engagement.ts
- `computeEngagementRate(likes, comments, followers)`:
  - Normal case: (100 + 5) / 2000 = 0.0525
  - followers = 0: should return 0 (not throw)
  - null likes/comments (Apify sparse data): treat as 0
- `detectOverperformers(posts[], threshold=1.5)`:
  - Empty array: returns []
  - < 30 posts: uses all as baseline, computes median, marks ≥1.5x
  - All posts same ER: no overperformers (nothing is 1.5x median if all equal)
  - Tie at boundary: higher absolute likes wins
  - Returns top 5 per account

### app/api/webhook/apify/route.ts
- Valid webhook, secret matches, posts inserted correctly
- Invalid/missing APIFY_WEBHOOK_SECRET header → 401
- Duplicate webhook (same apify_run_id) → ON CONFLICT DO NOTHING, returns 200
- Empty run result (0 posts from Apify) → noop, 200
- Sets scrape_runs.status='completed' on success

### app/api/feed/route.ts
- Returns posts sorted by scraped_at DESC
- Respects limit (max 20 per page)
- Unauthenticated request → 401

### app/api/save/route.ts
- Save card → saved_cards INSERT
- Unsave card → DELETE
- Save same card twice → idempotent (ON CONFLICT DO NOTHING)
- user_notes update → persisted correctly

## Integration Tests (Supabase local dev)

### RLS Isolation (CRITICAL)
Using `supabase test` or a Vitest integration test against `supabase start` local instance:

1. Create user_A and user_B
2. user_A creates niche_accounts [account_1]
3. Run scrape simulation: insert posts for account_1
4. Authenticate as user_B
5. SELECT * FROM posts → should return 0 rows
6. SELECT * FROM niche_accounts → should return 0 rows (user_B's own, not user_A's)
7. SELECT * FROM saved_cards WHERE user_id = user_A.id → should return 0 rows

This test must pass before production deployment. A failing RLS policy silently
exposes user data with no visible error.

### Queue Processor (stuck-job recovery)
1. Insert a scrape_queue item with status='processing' and processing_started_at = now() - 6 minutes
2. Run extract-post Edge Function
3. Verify: item gets reset to status='pending' (not picked up as a new job)
4. Run again: item gets processed normally

## E2E Tests (Playwright)

### Niche Setup Flow
- Visit /setup, add 3 Instagram handles
- Verify each appears in the account list
- Invalid handle (spaces, @@ syntax): validation error shown
- Submit with 0 handles: "Add at least one account" shown

### Feed State
- After niche setup (before first scrape): "Your feed is being built — check back Monday" shown
- After seeding posts via API: feed grid renders cards with hook_type, format, topic
- Click card → expands to "Recreate with your twist" template with editable fields
- Edit fields + save → user_notes persisted (verify via API)

### Playbook Flow
- Save a card from the feed → appears in /playbook immediately (optimistic)
- Unsave from /playbook → disappears immediately without reload
- Unsave from /feed → card remains in feed, disappears from /playbook

## Eval Tests (GPT-4o Extraction Quality)

### eval/extraction-accuracy.ts

Curate 20 labeled reels (caption + thumbnail + known hook_type/format/topic/cta):
- 5 with clear hook_type (curiosity gap, shock, contrast, story, question — 1 each)
- 5 with various formats
- 5 with unclear captions (tests degraded extraction)
- 5 with no thumbnail (text-only extraction path)

Run each through the OpenAI extraction prompt and compare against labels.
Target: >80% match on hook_type, >85% on format, >90% on topic angle (more lenient).

Run this eval:
- Before any prompt changes
- After any change to the extraction prompt
- As a gate before v1 launch

## Critical Paths

1. Weekly scrape cycle: cron fires → Apify runs → webhook received → posts inserted → queue populated → extracted → feed shows new cards Monday
2. RLS isolation: user A's posts are never visible to user B
3. Extraction accuracy: >80% hook_type match on 20-reel eval set
4. Stuck-job recovery: processing jobs older than 5 min get reset to pending

## Edge Cases to Verify

- Account with 0 posts or private account: Apify returns empty → no posts inserted, no error to user
- Thumbnail download fails: post still analyzed (text-only), thumbnail_storage_path = null, card shows placeholder image
- OpenAI API down: extraction fails, status='failed', retry on next 30s cycle
- Session expiry: redirect to /login on next request, no data loss
- User saves same card from 2 tabs simultaneously: ON CONFLICT DO NOTHING, no error
